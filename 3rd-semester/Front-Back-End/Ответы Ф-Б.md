## **5.1. Архитектура и Git (Базовые вопросы)**

## **№5. Три области Git: рабочая директория, индекс и репозиторий**

Git управляет версиями файлов, перемещая их между тремя основными логическими зонами. Понимание этого потока критически важно для работы с системой.

1. **Рабочая директория (Working Directory):** Это ваша текущая папка с проектом на диске (песочница). Здесь файлы находятся в их реальном, "видимом" состоянии, которое вы редактируете в IDE. В этой зоне файлы могут быть изменены, но Git еще не "запомнил" эти изменения.[youtube+1](https://www.youtube.com/watch?v=BIjrKuJGTxw)  
2. **Индекс (Staging Area / Index):** Это промежуточная область (буфер), файл-манифест, где формируется список изменений, готовых войти в следующий коммит. Это "черновик" будущего снимка системы. Вы выбираете конкретные изменения из рабочей директории и помещаете их сюда.[blogs.kenokivabe+1](https://blogs.kenokivabe.com/article/understanding-the-git-workflow-working-directory-staging-and-repository)  
3. **Репозиторий (Repository / .git):** Это база данных Git (хранится в скрытой папке `.git`), где навсегда сохранена вся история изменений в виде коммитов (снимков). Сюда попадают данные, которые были зафиксированы из индекса.\[[geeksforgeeks](https://www.geeksforgeeks.org/git/staging-in-git/)\]​

**Поток данных:**

* Изменения создаются в **Рабочей директории**.  
* Команда `git add` переносит их в **Индекс** (подготавливает к сохранению).  
* Команда `git commit` берет всё, что есть в **Индексе**, и сохраняет это в **Репозиторий** как новый снимок истории.\[[youtube](https://www.youtube.com/watch?v=ej-d8wVqVW4)\]​\[[blogs.kenokivabe](https://blogs.kenokivabe.com/article/understanding-the-git-workflow-working-directory-staging-and-repository)\]​

## **№6. Разница между командами `git add` и `git commit`**

Эти команды представляют собой два последовательных шага фиксации изменений. Разделение процесса на два этапа дает гибкость (можно разбить работу на логические части).

* **git add:** Эта команда не сохраняет историю. Она берет текущее состояние измененного файла из рабочей директории и обновляет запись о нем в **индексе** (Staging Area). Файл помечается как "staged" (подготовленный). Если вы измените файл после `git add`, эти новые изменения не попадут в коммит, пока вы снова не выполните `git add`.\[[youtube](https://www.youtube.com/watch?v=ej-d8wVqVW4)\]​\[[geeksforgeeks](https://www.geeksforgeeks.org/git/staging-in-git/)\]​  
* **git commit:** Эта команда берет все файлы, находящиеся в данный момент в **индексе**, и создает новый объект-коммит в **репозитории**. Она "цементирует" состояние проекта в истории, присваивая ему уникальный хеш и сообщение автора. После выполнения `git status` покажет, что "нечего коммитить" (working tree clean), так как индекс совпадает с последним коммитом.\[[blogs.kenokivabe](https://blogs.kenokivabe.com/article/understanding-the-git-workflow-working-directory-staging-and-repository)\]​

## **№3. Статические и динамические сайты**

Различие заключается в том, как контент доставляется пользователю.

1. **Статические сайты:**  
   * **Суть:** Сервер хранит готовые HTML-файлы и отдает их пользователю "как есть". Контент одинаков для всех посетителей.  
   * **Примеры:** Лендинг (посадочная страница), сайт-визитка, портфолио, документация.  
   * **Предпочтительны:** Когда важна максимальная скорость загрузки, дешевый хостинг, высокая безопасность (нет базы данных для взлома) и контент меняется редко.[bluehost+1](https://www.bluehost.com/blog/static-vs-dynamic-websites/)  
2. **Динамические сайты:**  
   * **Суть:** Страницы генерируются сервером "на лету" (server-side rendering) при каждом запросе, используя базы данных и скрипты (PHP, Python, Node.js). Контент подстраивается под пользователя.  
   * **Примеры:** Социальные сети (ВКонтакте), интернет-магазины (Ozon), форумы, банковские приложения.  
   * **Предпочтительны:** Когда нужна авторизация пользователей, персонализация, сложный функционал (корзина, комментарии) и частое обновление данных администраторами через CMS.\[[bluehost](https://www.bluehost.com/blog/static-vs-dynamic-websites/)\]​

## **№4. Сравнение монолитной и микросервисной архитектур**

| Характеристика | Монолитная архитектура | Микросервисная архитектура |
| :---- | :---- | :---- |
| **Суть** | Вся функциональность (UI, логика, работа с БД) упакована в одно единое приложение, работающее как один процесс. | Приложение разбито на набор небольших независимых сервисов, каждый из которых выполняет одну бизнес-задачу и общается с другими по сети (API). |

| **Недостатки** | При росте кодовой базы становится сложно поддерживать ("спагетти-код"). Падение одного модуля может обрушить весь сайт. | Сложность инфраструктуры (DevOps). Проблемы распределенных систем (сетевые задержки, транзакции, отладка) \[[fullscale](https://fullscale.io/blog/microservices-vs-monolithic-architecture/)\]​. |

## **№8. Конфликт слияния (Merge Conflict) в Git**

**Что это:** Ситуация, когда Git не может автоматически объединить изменения из двух веток, так как они противоречат друг другу. Чаще всего это происходит, когда два разработчика изменили одну и ту же строку в файле по-разному или один удалил файл, который другой редактировал.\[[izymes](https://izymes.com/2023/09/14/advanced-git-merge-conflict-resolution-techniques/)\]​

**Алгоритм решения:**

1. **Обнаружение:** При попытке `git merge` или `git pull` Git сообщит о конфликте (CONFLICT) и приостановит слияние.  
2. **Поиск файлов:** Команда `git status` покажет список файлов со статусом `both modified`.  
3. **Редактирование:** Открыть конфликтный файл в редакторе кода. Найти маркеры конфликта:  
   * `<<<<<<< HEAD` (ваши изменения)  
   * `=======` (разделитель)  
   * `>>>>>>> branch-name` (чужие изменения).  
     Вам нужно выбрать правильный вариант кода (свой, чужой или объединить оба) и удалить сами маркеры.  
4. **Фиксация:**  
   * Выполнить `git add <файл>`, чтобы пометить конфликт как решенный.  
   * Выполнить `git commit` для завершения слияния (обычно сообщение коммита генерируется автоматически).[geeksforgeeks+1](https://www.geeksforgeeks.org/git/merge-conflicts-and-how-to-handle-them/)

## **№2. Сравнение MPA и SPA приложений**

**Многостраничные приложения (MPA \- Multi-Page Application):**

* **Архитектура:** Классический подход. При каждом клике по ссылке браузер отправляет запрос на сервер, сервер генерирует *новую* HTML-страницу целиком и отправляет её обратно. Происходит полная перезагрузка страницы.  
* **Преимущества:** Отличная SEO-оптимизация "из коробки" (поисковики легко видят контент), меньше нагрузка на устройство пользователя (браузер), проще аналитика.  
* **Недостатки:** Медленнее для пользователя (белый экран при переходах), выше нагрузка на сервер.[wezom+1](https://wezom.com/blog/spa-vs-mpa-whats-best-for-your-project)

**Одностраничные приложения (SPA \- Single-Page Application):**

* **Архитектура:** Загружается один раз (один HTML-каркас). Далее при переходах JavaScript перехватывает клики, запрашивает у сервера только данные (JSON) и динамически перерисовывает часть страницы без перезагрузки.  
* **Преимущества:** Мгновенный отклик интерфейса ("ощущение нативного приложения"), экономия трафика (не качаем верстку повторно), возможность работы офлайн.  
* **Недостатки:** Сложности с SEO (нужен Server-Side Rendering), долгая первая загрузка (качается весь JS-бандл), сложнее в разработке (управление состоянием, утечки памяти).\[[wezom](https://wezom.com/blog/spa-vs-mpa-whats-best-for-your-project)\]​

## **5.2. HTML и основы верстки (Базовые вопросы)**

## **№11. Опорные узлы «скелета» HTML-документа**

Структура HTML-документа строится на вложенных друг в друга элементах, образующих «дерево».

* **`<!DOCTYPE html>`**: Это не тег, а инструкция браузеру, которая указывается на самой первой строке. Она сообщает, что документ написан по стандарту HTML5. Если её убрать, браузер может перейти в «режим совместимости» (quirks mode) и отображать страницу некорректно.  
* **`<html>`**: Корневой элемент всего документа. Все остальные теги (кроме doctype) должны находиться внутри него. Обычно имеет атрибут `lang` (например, `<html lang="ru">`), помогающий поисковикам и скринридерам определить язык контента.\[[hackfrontend](https://www.hackfrontend.com/docs/html-and-css/semantic-html)\]​  
* **`<head>`**: «Голова» документа. Содержит техническую информацию, которая **не отображается** напрямую на странице. Сюда помещают метатеги (кодировка, viewport), заголовок вкладки (`<title>`), ссылки на CSS-стили, шрифты и скрипты аналитики.\[[hackfrontend](https://www.hackfrontend.com/docs/html-and-css/semantic-html)\]​  
* **`<body>`**: «Тело» документа. Содержит весь видимый пользователю контент: текст, картинки, кнопки, видео, ссылки. Все, что мы видим в окне браузера, находится внутри этого тега.\[[hackfrontend](https://www.hackfrontend.com/docs/html-and-css/semantic-html)\]​

## **№12. Метатег viewport и адаптивная верстка**

Речь идет о теге:

xml  
`<meta name="viewport" content="width=device-width, initial-scale=1.0">`

* **Назначение:** Сообщает мобильному браузеру, как управлять размерами и масштабированием страницы. `width=device-width` говорит браузеру, что ширина области просмотра равна фактической ширине экрана устройства (а не стандартным для десктопа 980px). `initial-scale=1.0` устанавливает масштаб 1:1 при загрузке.  
* **Что будет без него:** Мобильные браузеры (iOS Safari, Android Chrome) по умолчанию считают, что сайт предназначен для больших мониторов. Они «сожмут» страницу (отзумят её), чтобы она поместилась на экран телефона целиком. Текст станет микроскопическим, пользователю придется приближать его пальцами.\[[hackfrontend](https://www.hackfrontend.com/docs/html-and-css/semantic-html)\]​

## **№13. Суть семантической разметки**

Семантическая верстка — это использование HTML-тегов по их смысловому назначению, а не только для внешнего вида. Вместо безликих `<div>` используются теги, говорящие «что это такое» (заголовок, статья, навигация). Это важно для SEO (поисковики лучше понимают контент) и доступности (скринридеры для незрячих могут корректно озвучить структуру).[rush-analytics+1](https://www.rush-analytics.ru/blog/semanticheskaya-verstka)

* **`<header>`**: Вводная часть страницы или раздела. Обычно содержит логотип, меню навигации, поиск.  
  * *Пример:* Верхняя шапка сайта с логотипом и кнопкой «Войти».\[[netpeak](https://netpeak.net/ru/blog/semanticheskaya-verstka-podrobnyy-gayd-i-vazhnost-dlya-seo/)\]​  
* **`<nav>`**: Блок навигации. Используется для главных меню, оглавлений, цепочек навигации («хлебных крошек»).  
  * *Пример:* Главное меню со ссылками «О нас», «Услуги», «Контакты».\[[netpeak](https://netpeak.net/ru/blog/semanticheskaya-verstka-podrobnyy-gayd-i-vazhnost-dlya-seo/)\]​  
* **`<main>`**: Основное уникальное содержимое страницы. На странице может быть только один `<main>`.  
  * *Пример:* Текст статьи, каталог товаров (без шапки и подвала).\[[netpeak](https://netpeak.net/ru/blog/semanticheskaya-verstka-podrobnyy-gayd-i-vazhnost-dlya-seo/)\]​  
* **`<footer>`**: «Подвал» сайта или раздела. Содержит копирайт, контакты, ссылки на соцсети, политику конфиденциальности.  
  * *Пример:* Нижняя черная полоса с надписью «© 2026 Компания».\[[netpeak](https://netpeak.net/ru/blog/semanticheskaya-verstka-podrobnyy-gayd-i-vazhnost-dlya-seo/)\]​

## **№14. Различия между `<article>`, `<section>` и `<div>`**

Эти теги часто путают, так как внешне они ведут себя как блоки, но семантически они имеют строгую иерархию.

| Тег | Суть | Пример контента |
| :---- | :---- | :---- |
| **`<article>`** | Независимый, самодостаточный контент. Если вырезать этот блок и вставить на другой сайт (или в RSS-ленту), он сохранит смысл. | Пост в блоге, новостная заметка, карточка товара в каталоге, отдельный комментарий пользователя [verstaem+1](https://verstaem.online/blog/semanticheskie-tegi-v-html/). |
| **`<section>`** | Тематическая группировка контента. Обязательно должна иметь заголовок (хотя бы скрытый). Является частью чего-то большего. | Глава в статье, блок «Наши преимущества» на лендинге, раздел «Отзывы» \[[verstaem](https://verstaem.online/blog/semanticheskie-tegi-v-html/)\]​. |
| **`<div>`** | Просто контейнер без смысла. Используется только для стилизации (CSS) или группировки элементов, когда семантические теги не подходят. | Обертка для центрирования контента (`wrapper`), декоративная полоска, сетка для layout \[[code-basics](https://code-basics.com/ru/languages/html/lessons/semantic-elements)\]​. |

## **№16. Типы ввода в формах и мобильные клавиатуры**

Атрибут `type` у тега `<input>` не только валидирует данные, но и кардинально меняет UX на мобильных устройствах, открывая подходящую виртуальную клавиатуру.

1. **`type="text"`**: Стандартная клавиатура.  
   * *Пример:* ФИО, адрес.  
2. **`type="email"`**: Клавиатура с латиницей, где на видном месте есть символы `@` и `.`.  
   * *Пример:* Форма подписки на рассылку.  
3. **`type="tel"`**: Цифровая клавиатура (numpad), часто с крупными кнопками цифр, плюсом и звездочкой.  
   * *Пример:* Ввод номера телефона для обратного звонка.  
4. **`type="number"`**: Цифровая клавиатура, иногда с точкой или запятой (зависит от ОС). Может добавлять стрелочки вверх/вниз в десктопных браузерах.  
   * *Пример:* Количество товара в корзине, возраст.  
5. **`type="url"`**: Клавиатура, где под рукой есть `.com`, `/` и `.`.

## **№21. Структура семантической страницы**

Пример правильной вложенности для типичной страницы блога:

xml  
`<body>`  
  `<!-- Шапка всего сайта -->`  
  `<header>`  
    `<img src="logo.png" alt="Логотип">`  
    `<nav> <!-- Главная навигация -->`  
      `<ul>`  
        `<li><a href="/">Главная</a></li>`  
        `<li><a href="/blog">Блог</a></li>`  
      `</ul>`  
    `</nav>`  
  `</header>`

  `<!-- Основной контент страницы -->`  
  `<main>`  
    `<h1>Последние новости</h1>`  
      
    `<!-- Секция со списком статей -->`  
    `<section>`  
      `<h2>Популярное за неделю</h2>`  
        
      `<!-- Карточка 1: Самостоятельная единица контента -->`  
      `<article>`  
        `<h3>Как учить HTML</h3>`  
        `<p>Краткое описание статьи...</p>`  
        `<a href="/post/1">Читать далее</a>`  
      `</article>`

      `<!-- Карточка 2 -->`  
      `<article>`  
        `<h3>Семантика для новичков</h3>`  
        `<p>Введение в теги...</p>`  
      `</article>`  
    `</section>`  
  `</main>`

  `<!-- Подвал сайта -->`  
  `<footer>`  
    `<p>&copy; 2026 Мой Блог</p>`  
  `</footer>`  
`</body>`

**Иерархия:**

1. **`header` и `footer`** обрамляют страницу сверху и снизу (сквозные элементы).  
2. **`main`** выделяет уникальный контент именно этой страницы (чтобы поисковик не индексировал шапку повторно).  
3. **`section`** группирует список новостей по теме.  
4. **`article`** используется для каждой новости, так как каждая новость — это целостный объект.[verstaem+1](https://verstaem.online/blog/semanticheskie-tegi-v-html/)

## **5.3. CSS и Bootstrap (Базовые вопросы)**

## **№22. Три способа подключения CSS к HTML-документу**

Существует три способа добавить стили на страницу. У них разные области применения и приоритеты.

1. **Внешние стили (External):** Подключение отдельного файла `.css`.  
   * **Как:** Тег `<link>` внутри `<head>`.  
   * **Пример:** `<link rel="stylesheet" href="style.css">`.  
   * **Когда использовать:** Основной способ для реальных проектов. Позволяет использовать один файл стилей для всего сайта.  
2. **Внутренние стили (Internal):** Стили пишутся внутри самого HTML-файла в «шапке».  
   * **Как:** Тег `<style>` внутри `<head>`.  
   * **Пример:**  
     xml

`<style>`  
  `body { background-color: #f0f0f0; }`  
`</style>`

*   
  * **Когда использовать:** Для уникальных стилей одной конкретной страницы (например, лендинг внутри большого сайта) или в email-рассылках.  
3. **Встроенные стили (Inline):** Стили пишутся прямо в HTML-теге.  
   * **Как:** Атрибут `style` у конкретного элемента.  
   * **Пример:** `<p style="color: red; font-size: 20px;">Текст</p>`.  
   * **Когда использовать:** Для быстрой отладки или динамического изменения стилей через JavaScript (например, анимация прогресс-бара).

**Приоритеты:**  
Если один и тот же элемент стилизуется разными способами, браузер выберет стиль с наивысшим приоритетом:

1. **Inline (Встроенные)** — самый высокий приоритет (перебивает всё, кроме `!important`).  
2. **Internal (Внутренние)** — обычно перебивает внешние стили (если тег `<style>` стоит ниже `<link>` в коде).  
3. **External (Внешние)** — базовый приоритет.[edu.sravni+1](https://edu.sravni.ru/kursy/info/podklyuchenie-css-k-html/)

## **№24. Блочная модель CSS (Box Model)**

Каждый элемент на веб\-странице — это прямоугольный «ящик». Блочная модель описывает структуру этого ящика и то, как рассчитывается его итоговый размер.

**Слои модели (от центра к краям):**

1. **Content (Контент):** Самая внутренняя часть — текст или картинка. Ее размеры задаются свойствами `width` и `height`.  
2. **Padding (Внутренний отступ):** Прозрачное пространство вокруг контента *внутри* рамки. Отдаляет текст от границы блока. Увеличивает видимый размер элемента.  
3. **Border (Граница):** Рамка вокруг padding. Имеет толщину, цвет и стиль. Тоже увеличивает размер блока.  
4. **Margin (Внешний отступ):** Прозрачное пространство *снаружи* рамки. Отталкивает соседние элементы друг от друга. Не влияет на размер самого элемента, но влияет на место, которое он занимает в потоке.

**Влияние на размер:**  
По умолчанию (свойство `box-sizing: content-box`), если вы задали `width: 100px`, `padding: 10px` и `border: 5px`, итоговая ширина блока на экране будет:  
`100 + 10*2 + 5*2 = 130px`.  
*Совет:* Чтобы ширина включала отступы и рамки (была ровно 100px), используют свойство `box-sizing: border-box`.

## **№25. Режимы отображения `display`: block, inline и inline-block**

| Свойство | Поведение | Примеры элементов |
| :---- | :---- | :---- |
| **Block** (Блочный) | Занимает всю доступную ширину родителя (растягивается на 100%). Всегда начинается с новой строки. Можно задавать ширину (`width`), высоту (`height`), любые отступы (`margin`, `padding`). | `<div>`, `<p>`, `<h1>`, `<header>`, `<ul>` |
| **Inline** (Строчный) | Занимает ровно столько места, сколько нужно контенту. Следующие элементы встают в ту же строку. **Нельзя** задать ширину и высоту. Вертикальные отступы (`margin-top/bottom`) не работают корректно (не отталкивают соседей). | `<span>`, `<a>`, `<b>`, `<strong>` |
| **Inline-block** (Блочно-строчный) | Гибрид. Внешне ведет себя как строчный (встает в строку с текстом), но внутри ведет себя как блок — ему **можно** задать точные размеры, рамки и любые отступы. | `<button>`, `<input>`, `<img>` |

## **№23. Специфичность (Вес) селекторов CSS**

Когда к одному элементу подходят несколько правил CSS (конфликт), браузер рассчитывает «вес» каждого селектора, чтобы выбрать победителя. Это похоже на спортивный счет из трех цифр (A-B-C).

**Система начисления баллов:**

* **ID селектор** (`#header`) \= 100 баллов.  
* **Класс, псевдокласс, атрибут** (`.menu`, `:hover`, `[type="text"]`) \= 10 баллов.  
* **Тег, псевдоэлемент** (`div`, `p`, `::before`) \= 1 балл.  
* **Универсальный селектор** (`*`) \= 0 баллов.  
* *Встроенный стиль (inline)* \= 1000 баллов (побеждает почти всех).  
* *\!important* \= «Ядерная кнопка», перебивает вообще всё (использовать с осторожностью).

**Пример:**

css  
*`/* Вес: 0-0-1 (Тег) */`*  
`p { color: red; }`

*`/* Вес: 0-1-0 (Класс) - ПОБЕДИТЕЛЬ */`*  
`.text { color: green; }`

Если у параграфа есть класс `text`, он будет зеленым, даже если правило `p` написано ниже.

## **№28. Flexbox: Основная идея и оси**

**Flexbox** (Flexible Box Layout) — это одномерная модель для раскладки элементов в ряд или в колонку. Она идеально подходит для выравнивания элементов внутри контейнера и распределения свободного места между ними.

**Ключевые понятия:**

1. **Главная ось (Main Axis):** Направление, вдоль которого выстраиваются элементы. По умолчанию — горизонтально слева направо (`flex-direction: row`).  
2. **Поперечная ось (Cross Axis):** Перпендикулярна главной. По умолчанию — вертикально сверху вниз.

**Свойства выравнивания:**

* **`justify-content`**: Управляет выравниванием вдоль **главной оси** (по горизонтали).  
  * *Примеры:* `center` (по центру), `space-between` (раскидать по краям), `flex-start` (прижать к началу).  
* **`align-items`**: Управляет выравниванием вдоль **поперечной оси** (по вертикали).  
  * *Примеры:* `center` (отцентровать по вертикали), `stretch` (растянуть на всю высоту), `flex-end` (прижать к низу).

## **№29. Основы CSS Grid Layout**

**Grid Layout** — это двумерная система сеток, позволяющая управлять расположением элементов одновременно и по строкам, и по столбцам (как таблица, но гибче).

* **Треки (Tracks):** Это полосы сетки — сами столбцы или строки. Пространство между двумя линиями сетки.  
* **Единица `fr` (fraction):** Гибкая единица измерения, означающая «доля свободного пространства».  
  * *Пример:* `grid-template-columns: 1fr 2fr;` создаст два столбца. Второй будет в два раза шире первого (займет 2/3 места, а первый — 1/3).  
* **Функция `repeat()`**: Сокращенная запись для повторяющихся треков.  
  * *Пример:* Вместо `1fr 1fr 1fr 1fr` можно написать `repeat(4, 1fr)`. Это создаст 4 одинаковых столбца.

**Пример использования:**

css  
`.container {`  
  `display: grid;`  
  `/* 3 колонки равной ширины */`  
  `grid-template-columns: repeat(3, 1fr);`  
  `/* Отступ между ячейками 20px */`  
  `gap: 20px;`  
`}`

## **1\. Форма с обязательными полями**

xml  
`<form action="/submit" method="post">`  
  `<div>`  
    `<label for="name">Ваше имя:</label>`  
    `<input type="text" id="name" name="name" required>`  
  `</div>`  
    
  `<div>`

**Объяснение:**

* **`required`**: Атрибут валидации, который запрещает отправку формы, если поле пустое. Браузер сам покажет сообщение "Заполните это поле".[dev+1](https://dev.to/chrisspotless/html-input-validation-attribute-1a30)  
* **`type="email"`**: Сообщает браузеру, что здесь ожидается email. На мобильных устройствах откроется клавиатура с символом `@`, а браузер проверит наличие этого символа перед отправкой.  
* **`for` \+ `id`**: Атрибут `for` у тега `<label>` должен совпадать с `id` инпута. Это создает программную связь: при клике на текст "Ваше имя" курсор автоматически встанет в поле ввода, что улучшает доступность (a11y).

## **2\. Центрирование контейнера**

css  
`.container {`  
  `max-width: 1200px;`  
  `width: 100%; /* Для адаптивности на экранах меньше 1200px */`  
  `margin: 0 auto;`  
`}`

**Объяснение:**

* **`max-width: 1200px`**: Ограничивает ширину блока. На больших экранах он не станет шире 1200px, а на маленьких (телефонах) будет занимать всю доступную ширину.  
* **`margin: 0 auto`**: Сокращенная запись `margin-top/bottom: 0` и `margin-left/right: auto`. Ключевое слово `auto` заставляет браузер автоматически рассчитать равные отступы слева и справа, помещая блок ровно по центру родителя.\[[selleo](https://selleo.com/blog/when-does-margin-0-auto-center)\]​

## **3\. Горизонтальное меню (Flexbox)**

css  
`.nav-menu {`  
  `display: flex;`  
  `justify-content: space-between;`  
  `list-style: none; /* Убрать точки списка */`  
  `padding: 0;`  
`}`

**Объяснение:**

* **`display: flex`**: Превращает контейнер во Flex-контейнер, выстраивая дочерние элементы (`li`) в ряд (по главной оси) по умолчанию.  
* **`justify-content: space-between`**: Распределяет свободное место между элементами. Первый элемент прижимается к левому краю, последний — к правому, а остальные равномерно распределяются между ними.\[[youtube](https://www.youtube.com/watch?v=2CjQxFIUBlA)\]​

## **4\. Отступы с помощью gap**

css  
`.container {`  
  `display: flex;`  
  `gap: 20px; /* Равные отступы */`  
`}`

**Объяснение:**

* **Преимущество `gap` перед `margin`**: Свойство `gap` создает отступы *только между* элементами. Вам не нужно писать сложные селекторы типа `li:not(:last-child) { margin-right: 20px }`, чтобы убрать лишний отступ у последнего элемента. Это чище и проще поддерживать.\[[youtube](https://www.youtube.com/watch?v=o273rdc4yi4)\]​

## **5\. Абсолютное позиционирование (Бейдж)**

css  
`.card {`  
  `position: relative; /* Точка отсчета для дочернего */`  
  `width: 200px;`  
  `height: 200px;`  
  `background: #eee;`  
`}`

`.badge {`  
  `position: absolute; /* Выпадает из потока */`  
  `top: 10px;          /* Отступ сверху */`  
  `right: 10px;        /* Отступ справа */`  
  `background: red;`  
  `color: white;`  
  `border-radius: 50%;`  
`}`

**Объяснение:**

* Элемент с `position: absolute` ищет ближайшего родителя с позиционированием отличным от `static`.  
* Задавая родителю `.card` свойство `position: relative`, мы делаем его "системой координат". Теперь координаты `top: 10px; right: 10px` для `.badge` будут отсчитываться от правого верхнего угла карточки, а не от окна браузера.\[[intellipaat](https://intellipaat.com/blog/how-to-set-position-absolute-but-relative-to-parent-in-css/)\]​

## **6\. Медиа-запрос для мобильных**

css  
`.container {`  
  `display: flex;`  
  `flex-direction: row; /* По умолчанию ряд */`  
`}`

`@media (max-width: 768px) {`  
  `.container {`  
    `flex-direction: column; /* Переключение в колонку */`  
  `}`  
`}`

**Объяснение:**

* Когда ширина экрана становится 768px или меньше (планшеты/телефоны), срабатывает правило внутри `@media`.  
* Свойство `flex-direction: column` перестраивает элементы вертикально друг под другом. Это нужно, чтобы на узких экранах контент не сжимался в нечитаемую "кашу", а удобно прокручивался вниз.

## **7\. Кнопка с анимацией (transition)**

css  
`.btn {`  
  `background-color: blue;`  
  `color: white;`  
  `padding: 10px 20px;`  
  `border: none;`  
  `cursor: pointer;`  
  `transition: all 0.3s ease; /* Плавность изменений */`  
`}`

`.btn:hover {`  
  `background-color: green;`  
  `transform: scale(1.05); /* Увеличение на 5% */`  
`}`

**Объяснение:**

* **`transition: all 0.3s`**: Говорит браузеру: "Любые изменения CSS-свойств этого элемента должны происходить не мгновенно, а плавно в течение 0.3 секунды".  
* Без этого свойства кнопка меняла бы цвет и размер рывком. `transform: scale(1.05)` — самый производительный способ менять размер без перерисовки (reflow) окружающего макета.

## **8\. Сетка Grid**

css  
`.grid-container {`  
  `display: grid;`  
  `grid-template-columns: repeat(4, 1fr);`  
  `gap: 20px;`  
`}`

**Объяснение:**

* **`grid-template-columns: repeat(4, 1fr)`**: Создает 4 колонки.

  * `1fr` (fraction) — одна доля свободного пространства. Это значит "раздели всю ширину на 4 равные части".  
* **`gap: 20px`**: Создает "дорожки" (пустое пространство) шириной 20px между ячейками сетки как по вертикали, так и по горизонтали.\[[mozilladevelopers.github](https://mozilladevelopers.github.io/playground/css-grid/04-fr-unit/)\]​

## **9\. Пропорции изображения (aspect-ratio)**

css  
`.img-container {`  
  `width: 100%;`  
  `aspect-ratio: 16 / 9;`  
  `background-color: #ccc; /* Заглушка пока грузится картинка */`  
  `overflow: hidden;`  
`}`

`.img-container img {`  
  `width: 100%;`  
  `height: 100%;`  
  `object-fit: cover;`  
`}`

**Объяснение:**

* **CLS (Cumulative Layout Shift)** — это когда текст "прыгает" вниз при подгрузке картинки, что раздражает пользователя.  
* **`aspect-ratio: 16 / 9`**: Браузер сразу резервирует место под картинку нужной высоты еще *до* того, как она загрузилась. Благодаря этому макет остается стабильным и не дергается.\[[youtube](https://www.youtube.com/watch?v=IYSizlSK858)\]​

## **11\. Адаптивная сетка**

xml  
`<div class="container">`  
  `<div class="row">`  
    `<!-- Классы складываются и применяются в зависимости от размера экрана -->`

**Объяснение:**

* Классы пишутся "на вырост", поэтому нет нужды писать `col-xs-12` (достаточно просто `col-12`), так как базовый стиль без префикса работает всегда, пока его не переопределит более "широкий" брейкпоинт.[stackoverflow+1](https://stackoverflow.com/questions/24175998/meaning-of-numbers-in-col-md-4-col-xs-1-col-lg-2-in-bootstrap)

## **16\. Компонент-счётчик с `useState`**

jsx  
`import React, { useState } from 'react';`

`function Counter() {`  
  `const [count, setCount] = useState(0);`

  `return (`  
    `<div>`  
      `<h1>Счёт: {count}</h1>`  
      `<button onClick={() => setCount(count + 1)}>+</button>`  
      `<button onClick={() => setCount(count - 1)}>−</button>`  
    `</div>`  
  `);`  
`}`

**Объяснение:**

* **Обновление состояния:** Функция `setCount` сообщает React, что данные изменились. React ставит это обновление в очередь. Важно: мы не изменяем переменную `count` напрямую (например, `count++`), а передаем *новое* значение.  
* **Перерисовка:** Когда состояние меняется, React запускает процесс **Reconciliation** (согласование). Он заново вызывает функцию компонента `Counter`, вычисляет новый результат JSX (Virtual DOM), сравнивает его с предыдущим и точечно обновляет реальный DOM в браузере (меняет только цифру в теге `<h1>`).

## **17\. Вывод списка с `map` и `key`**

jsx  
`function FrameworkList() {`  
  `const frameworks = [`  
    `{ id: 1, name: 'React' },`  
    `{ id: 2, name: 'Vue' }`  
  `];`

  `return (`  
    `<ul>`  
      `{frameworks.map((item) => (`  
        `<li key={item.id}>{item.name}</li>`  
      `))}`  
    `</ul>`  
  `);`  
`}`

**Объяснение:**

* **Роль `key`:** Атрибут `key` помогает алгоритмам React понять, какие элементы списка были изменены, добавлены или удалены.  
* **Почему уникальный и стабильный:**  
  * Если ключи не уникальны, React может перепутать элементы и неправильно обновить их состояние.  
  * Ключ должен быть **стабильным** (например, ID из базы данных), а не случайным (`Math.random()`) или просто индексом массива (если список сортируется). Если ключ изменится при следующем рендере, React уничтожит старый DOM-узел и создаст новый с нуля, что приведет к потере фокуса в инпутах и снижению производительности.

## **18\. Условный рендеринг**

jsx  
`function Page({ isLoading }) {`  
  `// Способ 1: Тернарный оператор (лучше для логики "ИЛИ-ИЛИ")`  
  `return (`  
    `<div>`  
      `{isLoading ? <Loader /> : <Content />}`  
    `</div>`  
  `);`

  `// Способ 2: Логическое И (&&) (обычно для "ПОКАЗАТЬ, ЕСЛИ...")`  
  `// Для реализации "else" с && потребуется две строки:`  
  `// {isLoading && <Loader />}`  
  `// {!isLoading && <Content />}`  
`}`

**Объяснение:**

* **Как это работает:** React умеет рендерить `null`, `false` или `undefined` как "ничего". Любое JavaScript-выражение внутри `{}` вычисляется.  
* **Тернарный оператор (`condition ? true : false`):** Если `isLoading` истинно, возвращается первый компонент, иначе — второй. Это стандарт для переключения состояний.  
* **Оператор `&&`:** Если левая часть (`isLoading`) правдива, React рендерит правую часть. Если ложна — React останавливается и ничего не выводит.

## **19\. Управляемое текстовое поле (Controlled Input)**

jsx  
`function TextInput() {`  
  `const [text, setText] = useState('');`

  `const handleChange = (event) => {`  
    `setText(event.target.value); // 2. Обновляем стейт`  
  `};`

  `return (`  
    `// 1. Значение поля жестко привязано к стейту`  
    `<input`   
      `type="text"`   
      `value={text}`   
      `onChange={handleChange}`   
    `/>`  
  `);`  
`}`

**Объяснение:**

* **Суть "Управляемого компонента":** В обычном HTML инпут сам хранит своё значение. В React мы отбираем это право. Теперь **"источником истины"** является состояние (`state`) React.  
* **Процесс:**  
  1. Пользователь нажимает клавишу.  
  2. Срабатывает `onChange`.  
  3. Мы вызываем `setText`, обновляя React-state.  
  4. React перерисовывает компонент и передает новое значение обратно в атрибут `value` инпута.  
     Это позволяет мгновенно валидировать ввод или форматировать его на лету.

## **20\. `useEffect` при монтировании**

jsx  
`import React, { useEffect } from 'react';`

`function App() {`  
  `useEffect(() => {`  
    `console.log('Компонент смонтирован (Mounted)');`  
      
    `// Опционально: функция очистки (вызовется при размонтировании)`  
    `return () => console.log('Unmounted');`  
  `}, []); // <--- Пустой массив зависимостей`

  `return <div>Hello World</div>;`  
`}`

**Объяснение:**

* **Роль `[]`:** Второй аргумент `useEffect` — это массив зависимостей. React следит за переменными в этом массиве. Если они меняются, эффект запускается снова.  
* **Пустой массив:** Означает "зависимостей нет". Следовательно, эффект запустится **только один раз** сразу после первого рендера (монтирования). Это аналог устаревшего метода `componentDidMount`.  
* **Если убрать массив:** `useEffect(() => { ... })` (без второго аргумента) будет запускаться **после каждого рендера** (при любом обновлении состояния или пропсов), что может вызвать бесконечный цикл или падение производительности.

## **21\. Реактивный счётчик с `ref`**

xml  
`<script setup>`  
`import { ref } from 'vue'`

`const count = ref(0)`

`const increment = () => {`  
  `count.value++ // В скрипте обращаемся через .value`  
`}`  
`</script>`

`<template>`  
  `<!-- В шаблоне .value не нужен, Vue "разворачивает" ref автоматически -->`  
  `<button @click="increment">Счёт: {{ count }}</button>`  
`</template>`

**Объяснение:**

* **Доступ к значению:** Функция `ref()` оборачивает примитив (число 0\) в объект-прокси. Внутри `<script>` мы обязаны обращаться к значению через свойство `.value`. Однако в `<template>` Vue автоматически "разворачивает" (unwrap) ref-объекты, поэтому мы пишем просто `{{ count }}`.  
* **Отслеживание:** Vue использует систему реактивности на основе Javascript Proxy. Когда мы меняем `.value`, Vue перехватывает это изменение (trap) и автоматически запускает перерисовку всех компонентов, где используется эта переменная.

## **22\. Список с `v-for` и `:key`**

xml  
`<script setup>`  
`import { ref } from 'vue'`

`const frameworks = ref([`  
  `{ id: 1, title: 'Vue' },`  
  `{ id: 2, title: 'React' }`  
`])`  
`</script>`

`<template>`  
  `<ul>`  
    `<li v-for="item in frameworks" :key="item.id">`  
      `{{ item.title }}`  
    `</li>`  
  `</ul>`  
`</template>`

**Объяснение:**

* **Зачем нужен `:key`:** Это уникальный идентификатор для алгоритма Virtual DOM. Он помогает Vue понять, какой именно элемент списка изменился, переместился или удалился.  
* **Если не указать:** Vue будет использовать стратегию "patch-in-place" (обновление на месте). Он попытается переиспользовать существующие DOM-элементы, просто меняя в них текст. Это быстрее, но может привести к критическим багам, если элементы имеют собственное состояние (например, фокус в инпуте, анимации или выделенный текст), так как состояние останется на месте, а данные поменяются.

## **23\. Вычисляемое свойство `computed`**

xml  
`<script setup>`  
`import { ref, computed } from 'vue'`

`const tasks = ref([`  
  `{ id: 1, text: 'Купить хлеб', completed: true },`  
  `{ id: 2, text: 'Выучить Vue', completed: false }`  
`])`

`// Вычисляемое свойство`  
`const completedTasks = computed(() => {`  
  `return tasks.value.filter(task => task.completed)`  
`})`  
`</script>`

**Объяснение:**

* **Преимущество (Кеширование):** `computed` свойство — это "умная" переменная. Она **кеширует** результат. Если вы обратитесь к `completedTasks` 100 раз подряд, функция-фильтр выполнится только 1 раз. Обычный метод (`function`) выполнялся бы заново при каждой перерисовке страницы.  
* **Пересчет:** Значение пересчитывается **только тогда**, когда меняются его реактивные зависимости (в данном случае массив `tasks`).

## **24\. Двусторонняя привязка `v-model`**

xml  
`<script setup>`  
`import { ref } from 'vue'`

`const searchQuery = ref('')`  
`</script>`

`<template>`  
  `<input type="text" v-model="searchQuery" placeholder="Поиск...">`  
  `<p>Вы ищете: {{ searchQuery }}</p>`  
`</template>`

**Объяснение:**

* **Двусторонняя привязка:** Это "синтаксический сахар".  
  1. **View \-\> Model:** Когда пользователь печатает в `input`, Vue автоматически обновляет переменную `searchQuery` (через событие `input`).  
  2. **Model \-\> View:** Если мы изменим `searchQuery` программно (например, кнопкой "Очистить"), текст в инпуте тоже обновится (через атрибут `value`).  
* По сути, `v-model` разворачивается в: `:value="searchQuery" @input="searchQuery = $event.target.value"`.
